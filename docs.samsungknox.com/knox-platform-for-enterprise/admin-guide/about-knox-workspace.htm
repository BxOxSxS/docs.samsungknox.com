Knox Workspace

On this page

The Workspace is a secure container designed to separate, isolate, encrypt, and protect enterprise apps and data. This container enables a mobile phone that provides both work and play environments. Enterprises can remotely manage the work environment, but not access personal information like photos and messages.

How Workspace operates

Knox is a super set of features built on top the Android Enterprise framework. Workspace is a Hardware-based security solution that has been built on the Knox Platform using ARM TrustZone, it only works on Samsung devices.

In an Enterprise deployment, Google provides three modes of Android Enterprise, Managed Device, Work Profile and Fully Managed with Work Profile.

  • Work Profile: Helps admins manage the BYOD (Bring Your Own Device) use-case, where an device user owns the device and uses it both personally and for work. The agent sits inside the container area as Profile Owner (PO) to separate work apps from personal apps. IT admins can control the work area only, and have visibility over the personal area.
  • Managed Device: Helps admins manage CL (Corporate Liable) devices that is owned by the enterprise. When a device is enrolled as Managed Device, IT admins have full control over the device; the agent sits as Device Owner (DO) of the device.
  • Fully managed Device with Work Profile: Enterprises have full control of the device. This is typically used as a single purpose device, for example, like a work kiosk at a restaurant.

Knox Workspace is the name of Work Profile with a Knox License. It is supported in the following upgrade paths.

  • Work Profile > (Knox License) > Workspace.
  • Fully managed with Work Profile > (Knox License) > Workspace on fully managed device.

The apps and data inside Workspace are isolated from those outside the Workspace. Specifically, apps outside the Workspace can't use Android inter-process communication with apps inside the Workspace. Data sharing is also limited. For example, photos taken with the camera inside the Workspace are not viewable in the Gallery outside the Workspace. The same restriction applies to copying and pasting. When allowed by an IT policy, some application data, such as contacts and calendar data, can be shared across the Workspace boundary. The device user can choose whether to share contacts and calendar notes between the Workspace and personal space. However, an IT policy ultimately controls this option.

An enterprise can manage the Workspace like any other device feature, using an MDM system. Samsung Knox includes a large set of policies for authentication, data security, VPN, e-mail, application blacklisting, whitelisting and so on.

The following images show the device's home screen with the personal side (left) and the Workspace home screen after the user has logged in (right). Apps inside the Workspace are distinguished by a keyhole badge in the lower right corner of their launcher icons.

Use case

  • A small start-up company supports a BYOD program, allowing employees to use their personal phones for work.
  • To isolate the apps managing confidential information, the company uses a Knox Workspace container on each personal phone.
  • The IT admin uses an MDM console to create a container, which has pre-installed email and browser apps already in the container for work.

Benefits

  • Prevention of apps intentionally or inadvertently accessing apps or data they are not authorized to.
  • Protection of private data, which is managed in a sand-boxed environment.
  • Easy separation of personal apps and business apps, which are in a Knox Workspace container.
For a full overview of how Knox differs to Android Enterprise, read the Knox White Paper.

Policy changes

The following features are affected when you upgrade an Android Work Profile to a Workspace 3.x.

For more information on the new Workspace 3.x container architecture, see Knox 3.0 SDK: new container architecture.
Feature Change
Badges

Applicable to Workspace only

Upon Knox license activation, the Workspace and app badge changes from a briefcase to a shield to indicate that the Knox license has been activated.
Password Enforcement

Applicable to Workspace only

the Knox framework doesn't enforce passwords by default. IT admins can leverage either Android policies or Knox policies to apply password policies based on their enterprise requirements.

  • Android password policy: See your MDM console for more details.
  • Knox password policy: See your MDM console for more details.
Default Policies

The Knox framework doesn't enforce stricter defaults for either the device or the configuration profile. The following features are enabled by default. IT admin can call Android or Knox policies to lock down the device or the configuration profile:

  • USB Debugging
  • Screen Capture
  • Bluetooth
  • NFC
  • Copy and Paste from profile to device
Default Apps

Android enables the following apps by default for the Android Work profile. Knox doesn't enable additional apps by default for profile:

  • Contacts
  • My Files
  • PlayStore
Work Settings

Applicable to Workspace profile only

  • In Android N, Workspace settings are part of the device settings.
  • In Android O, Workspace settings are available to the user as a separate app inside the profile.
  • In Android P, Workspace settings are in the same menu as the device settings.
Screen Capture

Prior to Knox license activation, Android Work screen shots are stored on the device side.

After a Knox License activation, screen shots are stored inside the Workspace.
Keyboard and Input Methods

Knox 3.2.1: The Knox framework starting from Knox 3.2.1 separates personal and profile keyboards to ensure that important work data is not compromised. Admins do not have to whitelist IMEs any more.

Knox 3.0 - 3.2: Only one IME is allowed for the entire device. Both device and profile have one IME. System IME is the default. However, IT admin can whitelis third party IMEs as well. 

Knox 2.9 and lower (Knox Workspace CL/B2B and COM modes): Have one separate IME per user.
Approved Installers By default, Google Play is the only approved installer. IT admins can whitelist other apps as installers as well.
Biometric Authentication Management of biometric authentication (fingerprint and iris) are separate for Workspaces than the personal side. End user registers biometrics separately for profile. End user manages (add, remove and update) profile specific biometric data via work settings.
Contacts and Calendar

Applicable to Workspace only

Android O and below: the device user is able to search work contacts from personal side. The device user is also able to import and export work contacts and calendar. The IT admin can enable and disable contact search the import and export of work contacts and calendar.

Android P: the device user is no longer able to import and export work contacts and calendar.
Number of allowed containers

Only one B2B container and one personal container can be created on the device. Devices can have only one Workspace and only one Secure Folder. Two Workspaces are not allowed.
Policy Conflict

Now that a Device Owner and Profile Owner can call both Knox and Android policies on the same solution there are certain policies that will conflict.

  • Exact Match polices : Knox will eventually deprecate policies that are exactly the same as the Android counterpart. Our recommendation is to use Android policies.
  • Partial Match or Superset: For policies that match partially or are superset of the other the recommendation, we do NOT recommenced to mix and match policies. You should either call Knox policies entirely OR Android policies entirely.
Policies moved to User Scope

Certain policies that are container scope (can only be called on the container) are moved to user scope. These policies can be called by the primary user (Device Owner) or the secondary user (Profile Owner). The policies are only applicable to the user that calls them.

For example, "enforce multi-factor authentication" was previously container scope (Knox 2.9) and is now user scope (Knox 3.2.1). Thus, the Device Owner can lock the device or enforce multi-factor authentication for the entire device with a single MDM policy.
