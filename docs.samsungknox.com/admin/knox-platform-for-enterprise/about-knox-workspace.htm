Containers

The Android Enterprise work profile is a secure container that separates work apps and data from personal apps and data. Work profiles allow an IT department to securely manage a work environment without restricting users from using their device for personal apps and data.

Knox Platform for Enterprise is a superset of features built on top the Android Enterprise framework. On Samsung devices, a work profile is a hardware-based security solution built on the secure ARM TrustZone. Through Knox Platform for Enterprise, IT admins have additional, more granular controls over work profile on Samsung devices.

How it works

In an enterprise deployment, Google provides three modes of Android Enterprise:

  • work profile: Helps admins manage the BYOD (Bring Your Own Device) use-case, where a device user owns the device and uses it both personally and for work. A UEM app sits inside the work profile as the Profile Owner (PO), providing IT admins with control over the work profile only.
  • fully managed device: Helps admins manage company-owned devices that are used for work only. A UEM app runs as the Device Owner (DO) of the device, providing IT admins with full control over the device.
  • fully managed device with work profile: Helps admins manage company-owned devices that are used for both work and personal purposes. A UEM app runs as both a Device Owner as well as a Profile Owner, providing IT admins with full control over both.

The apps and data inside a work profile are isolated from those outside the work profile. Specifically, apps outside the work profile can't use Android inter-process communication with apps inside the work profile. Data sharing is also limited. For example, photos taken with the camera inside the work profile are not viewable in the Gallery outside the work profile. The same restriction applies to copying and pasting. When allowed by an IT policy, some app data, such as contacts and calendar data, can be shared across the work profile boundary. The device user can choose whether to share contacts and calendar notes between the work profile and personal space. However, an IT policy ultimately controls this option.

An enterprise can manage the work profile like any other device feature, using a UEM system. The Knox Platform for Enterprise offers a large set of policies for authentication, data security, VPN, e-mail, application blacklisting, whitelisting, and so on.

The following images show the device's home screen with the personal side (left) and the work profile home screen after the user has logged in (right). Apps inside the work profile are distinguished by a keyhole badge in the lower right corner of their launcher icons.

Use case

  • A small start-up company supports a BYOD program, allowing employees to use their personal phones for work.
  • To isolate the apps managing confidential information, the company uses a Knox Workspace container on each personal phone.
  • The IT admin uses an MDM console to create a container, which has pre-installed email and browser apps already in the container for work.

Benefits

  • Prevention of apps intentionally or inadvertently accessing apps or data they are not authorized to.
  • Protection of private data, which is managed in a sand-boxed environment.
  • Easy separation of personal apps and business apps, which are in a Knox Workspace container.
For more information on Knox features, read the Knox White Paper.

Policy changes

The following features are affected when you upgrade to a work profile in Knox 3.x.

FeatureChange
Badges

Applicable to work profile only

Upon Knox license activation, the work profile and app badge changes from a briefcase to a shield to indicate that the Knox license has been activated.

Password Enforcement

Applicable to work profile only

The Knox framework doesn't enforce passwords by default. IT admins can leverage either Android policies or Knox policies to apply password policies based on their enterprise requirements.

  • Android password policy: See your UEM console for more details.
  • Knox password policy: See your UEM console for more details.
Default Policies

The Knox framework doesn't enforce stricter defaults for either the device or the configuration profile. The following features are enabled by default. IT admins can deploy Android or Knox policies to lock down the device or the configuration profile:

  • USB Debugging
  • Screen Capture
  • Bluetooth
  • NFC
  • Copy and Paste from profile to device
Default Apps

Android enables the following apps by default for the work profile. Knox doesn't enable additional apps by default for profile:

  • Contacts
  • My Files
  • PlayStore
Work Settings

Applicable to work profile only

  • In Android N, work profile settings are part of the device settings.
  • In Android O, work profile settings are available to the user as a separate app inside the profile.
  • In Android P, work profile settings are in the same menu as the device settings.
Screen Capture

Prior to Knox license activation, work profile screen shots are stored on the device side.

After a Knox License activation, screen shots are stored inside the work profile.

Keyboard and Input Methods

Knox 3.2.1: The Knox framework starting from Knox 3.2.1 separates personal and profile keyboards to ensure that important work data is not compromised. Admins do not have to whitelist IMEs any more.

Knox 3.0 - 3.2: Only one IME is allowed for the entire device. Both device and profile have one IME. System IME is the default. However, IT admin can whitelist third party IMEs as well. 

Knox 2.9 and lower: (Legacy Knox Workspace CL and COM modes) Have one separate IME per user.

Approved InstallersBy default, Google Play is the only approved installer. IT admins can whitelist other apps as installers as well.
Biometric AuthenticationManagement of biometric authentication (fingerprint and iris) are separate for Workspaces than the personal side. End user registers biometrics separately for profile. End user manages (add, remove and update) profile specific biometric data via work settings.
Contacts and Calendar

Applicable to work profile only

Android O and below: The device user is able to search work contacts from personal side. The device user is also able to import and export work contacts and calendar. The IT admin can enable and disable contact search the import and export of work contacts and calendar.

Android P: The device user is no longer able to import and export work contacts and calendar.

Number of allowed containers

Only one B2B container and one personal container can be created on the device. Devices can have only one work profile and only one Secure Folder. Two work profiles are not allowed.

Policy Conflict

Now that a Device Owner and Profile Owner can call both Knox and Android policies on the same solution there are certain policies that will conflict.

  • Exact Match polices : Knox will eventually deprecate policies that are exactly the same as the Android counterpart. Our recommendation is to use Android policies.
  • Partial Match or Superset: For policies that match partially or are superset of the other the recommendation, we do NOT recommenced to mix and match policies. You should either call Knox policies entirely OR Android policies entirely.
Policies moved to User Scope

Certain policies that are container scope (can only be called on the container) are moved to user scope. These policies can be called by the primary user (Device Owner) or the secondary user (Profile Owner). The policies are only applicable to the user that calls them.

For example, "enforce multi-factor authentication" was previously container scope (Knox 2.9) and is now user scope (Knox 3.2.1). Thus, the Device Owner can lock the device or enforce multi-factor authentication for the entire device with a single policy.
