About the Knox Webhook Notification API

The Knox Webhook Notification (KWN) API is based on a webhook mechanism that allows different Knox cloud services to send change notifications to a subscribed client when a subscribed event or background process is complete.

Since most client requests are resource-intensive, KWN offers the advantage of asynchronous APIs that run these client events as background processes, while freeing up resources for other client events. Such a resource-efficient API workflow allows multiple requests to be handled simultaneously, without any inefficient event dependencies.

To leverage this notification service, a client must first subscribe to the KWN Subscription API to receive change notifications whenever an event trigger occurs.

Knox Webhook Notification API workflow

The following procedure summarizes the Knox Webhook Notification API workflow:

  1. The client downloads the Samsung Knox validation certificate for message signing and to validate the response from KWN.
  2. The client uses Knox Cloud Authentication API access tokens to call different Knox cloud service APIs. Within the KWN workflow, you are the API client that wants to receive change notifications.
  3. The client registers to an event with the KWN Subscription API. For example, an event could be uploading 10,000 devices using the Knox E-FOTA bulk device upload operation, or enrolling a device using the Knox Guard device upload operation.
  4. The event is now subscribed. A database stores the event subscription and the associated event.
  5. The client makes a call to a Knox cloud services API. Currently, KWN can be used with the following Knox cloud service APIs:
    • Knox E-FOTA
    • Knox Guard
  6. The completed Knox cloud services operation, through the Knox cloud service APIs, triggers an asynchronous event.
  7. KWN receives the asynchronous event from the Knox cloud services API operation. Then, KWN performs a database lookup to get the configured callback URL.
  8. The client receives the KWN callback response along with the signature for validation.
  9. The client validates the authenticity of the callback response by comparing the signature against the public key contained in the Samsung Knox certificate.
  10. Successful validation marks the completion of your request.

NOTE — Steps 1, 2, and 3 need to be completed only once. However, if your Samsung Knox validation certificate expires, you need to reperform Step 1.
